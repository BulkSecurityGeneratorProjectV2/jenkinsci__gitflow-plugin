Gitflow

Der Gitflow von Vincent Driessen

http://nvie.com/posts/a-successful-git-branching-model

  • Hat sich zum Quasi-Standard für Gitprojekte entwicklelt - mit Variationen
  • Ermöglicht parallele Entwicklung von Releases, Bugfixes und Hotfixes
  • Sehr gut geeignet für Semantic Versioning
  • vs. Trunk-basierte Entwicklung

Maven

  • Snapshots & Releases
  • Releases per Maven Release Plugin
  • Trunk-basierte Entwicklung
  • Versionierung: Snapshots & Releases
  • Snapshots: Inhalte sind nicht definiert, Änderungen unterworfen
  • Releases: Inhalte sind definiert, Änderungen sind (quasi) nicht möglich
  • Bei Einführung des Maven Release Plugins war Gitflow noch unbekannt
  • Maven Release Plugin benötigt (quasi) einen festen Branch
  • (Quasi) nur im trunk-basierten Ansatz einsetzbar
  • Also gibt es einen Widerspruch zum Gitflow

Features

  • Zeigt auch die Initialisierung (mit Version 1.0.0-SNAPSHOT)
  • Features werden auf dem Development-Branch entwickelt
  • Oder in Feature-Branches
  • Auch parallel/konkurrierend
  • Aber: Feature-Entwicklung ist nicht unser Thema

Start Release

  1. Wenn Feature-Complete
  2. Release-Branch erstellen
  3. Release-Version in den Maven-POMs setzen ("-Snapshot abhängen")
  4. Release Tag erstellen (im Release-Branch, nicht im master-Branch)
  5. Jetzt wird auch das Release gebaut und in den Maven-Repository-Manager deployt
  6. Die Entwicklungsversion für die kommenden Release-Patches/-Bugfixes in den Maven-POMs setzen
  7. Im develop-Branch die Entwicklungsversion für das nächste Release in den Maven-POMs setzen
  8. Die Release-Version wird auf dem Test-/Staging-System installiert

Test Release

  1. Zusätzliche Aktion (gegenüber dem Standard-Gitflow)
  2. Bugfix-Iteration(en)
  3. Alle gemeldeten Bugfixes implementieren
  4. Bugfixes möglichst gleich per Cherry-Pick in den develop-Branch überführen
  5. Cherry-Picks ggf. komplexer wg. Bugfix-Feature-Flow, Pull Request u.ä. - führt hier (zeitlich) zu weit
  6. Wenn alle gemeldeten Bugfixes gelöst sind:
  7. Patch-Release-Version in den Maven-POMs setzen ("-Snapshot abhängen")
  8. Patch-Release Tag erstellen
  9. Jetzt wird das Patch-Release gebaut und in den Maven-Repository-Manager deployt
  10. Die Entwicklungsversion für die nächste Bugfix-Iteration in den Maven-POMs setzen
  11. Die Patch-Release-Version wird auf dem Test-/Staging-System installiert

Publish Release

  1. Wenn kein weiterer Bug gemeldet wird
  2. Non-Fast-Forward-Merge des letzten Release-Commits auf den master
  3. Wichtig: nicht den nächsten DEV-Commt (SNAPSHOT) mergen!
  4. Im master ist imm das letzte stabile Release
  5. Ggf. auch Merge auf develop: Variante zum Cherry-Pick für die Bugfixes (aber nicht so gut)
  6. Patch-Release Tag erstellen

Finish Release

  1. Löschen des Release-Branches
  2. Abhängig vom Entwicklungs- und Release-Workflow
  3. Ggf. kann der Branch weiter für Bugfixes verwendet werden
  4. Ansonsten gibt es ja noch Hotfixes

Hotfixes

  1. Für Hotfixes am letzten publizierten Release
  2. Man könnte stattdessen auch den Release-Branch weiter laufen lassen
  3. Analog zu den Release-Aktionen
  4. Start Release setzt lediglich die Entwicklungsversion für den Hotfix (redundant, aber nötig, wenn der Release-Branch gelöscht wurde)
  5. Test, Publish und Finish Hotfix analog zu Releases
  6. Der Hotfix-Commit sollte eher nicht gemergt werden, Cherry-Picks sind hier zu 99% das Mittel der Wahl

Jenkins Gitflow Plugin

  • Automatisierung der vielen manuellen Schritte
  • Steuerung mittels grafischer Benutzeroberfläche
  • Gründe für die Umsetzung als Jenkins-Plugin:
    • Erweiterung des Maven Release Plugins sehr aufwändig, kaum vernünftig möglich
    • Verbereitung von Jenkins,
      Einsatzhäufigkeit bei Kunden
    • Build diverser Branches in einem Job
  • Es gibt/gab kein Werkzeug, das Gitflow und Maven zusammen bringt
  • Keine unterschiedlichen Jobs je Branch erforderlich
